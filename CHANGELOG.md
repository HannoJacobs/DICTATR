# Changelog

Changelog policy:
- Current known-status caveat: as of `2026-04-13 18:54 SAST`, the installed current build has now been exercised successfully on real AirPods and Bose QC45 Bluetooth dictation paths. The shipped notes should say the new start/readiness model appears to have worked on those retests, while still avoiding a blanket claim that every Bluetooth headset or every future route-recovery edge case is solved forever.
- Every shipped version entry must be written as a detailed operator-facing record, not a terse marketing summary.
- The current release entry is required to be very verbose: the release pipeline now fails if the section for the bundled app version has fewer than 8 bullet points or fewer than 1200 characters.

## 2.0
- Promoted the bundled app version from `1.31` to `2.0` because this is not a minor polish pass; it is the first shipped attempt that materially changes DICTATR’s Bluetooth recorder start model for the long-running reconnect-loop defect that had been the main unresolved product bug for days. The version bump is meant to signal that the app’s recorder behavior, release notes, and operator expectations changed in a substantial way.
- Kept the recorder fix centered on one generic principle instead of per-headset hacks: DICTATR no longer treats a single pre-start `AVAudioEngine.inputNode.outputFormat(forBus: 0)` snapshot as sufficient truth on Bluetooth routes. The recorder now waits for a live graph where the input and output formats are both non-zero and equal before it treats the route as ready enough to install the tap and commit the session.
- Preserved the startup-deadline model introduced in the previous patch so the app cannot sit in a fake recording state forever. A Bluetooth attempt must either reach a live graph-ready state or produce its first callback before the deadline, otherwise it is classified as a recoverable start failure instead of being allowed to masquerade as an active recording with zero real capture behind it.
- Preserved the narrower configuration-change handling model as well: DICTATR no longer treats “Bluetooth is involved” as automatic grounds for a fatal reset while the engine is still alive. On a live engine the app now rereads the graph formats and reinstalls the tap only when the graph is still genuinely ready, which is the behavior the April 13 evidence actually supported.
- Carried forward the non-default `AVCaptureSession + AVCaptureAudioDataOutput` comparison backend and the recorder-start policy tests. The point of keeping both in the shipped tree is that a future fallback or backend switch does not need to restart the entire investigation if more Bluetooth hardware surfaces a failure that the repaired `AVAudioEngine` path still cannot handle.
- Added concrete installed-build AirPods verification to the release record. In [`dictatr-20260413-175125-2e38e948.log`](/Users/hannojacobs/Library/Application%20Support/DICTATR/Logs/dictatr-20260413-175125-2e38e948.log), AirPods session `c89d634e` started from a mixed `24000/48000` fingerprint, reached `post_start_format_ready` with a live `24000/24000` graph, wrote `84630` frames with `captureState=flowing`, and returned the dictated text successfully from `/Applications/DICTATR.app`.
- Added concrete installed-build Bose QC45 verification to the release record. In [`dictatr-20260413-185034-bd0e09bb.log`](/Users/hannojacobs/Library/Application%20Support/DICTATR/Logs/dictatr-20260413-185034-bd0e09bb.log), Bose session `f03ba0c5` started from a mixed `16000/44100` fingerprint at line `77`, reached `post_start_format_ready` with a live `16000/16000` graph at line `80`, stayed in `captureState=flowing`, wrote `45056` frames at line `97`, and transcribed `"Testing, testing 12345."` at line `106`.
- Fixed the GitHub Actions DMG workflow so CI no longer depends on the locally generated `.swiftpm/xcode/package.xcworkspace` path that does not reliably exist on a fresh runner checkout. `create-dmg.sh` now falls back to archiving the package scheme directly from the repo root when that generated workspace is absent, which matches the invocation style that already works in clean environments.

CI now validates packaging and uploads the DMG as a workflow artifact instead of mutating release history behind the full-send flow.
- The correct shipped framing for `2.0` is therefore narrower and stronger at the same time: the repaired recorder start/readiness model now appears to have fixed the reproduced April 13 Bluetooth reconnect-loop case on both AirPods and Bose QC45 retests from the installed app, and the CI packaging workflow no longer fails on the generated-workspace assumption. Broader headset coverage is still an operational verification task, but `2.0` should be treated as the first release where the lingering Bluetooth start bug looks materially resolved rather than merely instrumented.

## 1.31
- Reworked DICTATR’s Bluetooth recorder startup so the app no longer commits to a tap from a single pre-start `AVAudioEngine.inputNode.outputFormat(forBus: 0)` snapshot. The recorder now logs both `inputNode.inputFormat(forBus: 0)` and `outputFormat(forBus: 0)` before and after `engine.start()`, attaches the configuration-change observer before `engine.start()`, and only installs the tap from a live post-start graph when the input and output formats are both non-zero and equal. This directly targets the April 13 AirPods failure signature where the route already showed `defaultInput=24000` while the preflight engine format was still stale `48000`.
- Added an explicit startup-deadline path keyed off the first capture callback instead of route quiet. A new recording or retry no longer counts as “good” just because the route has been quiet for a while; it must either reach a live graph-ready state or produce the first callback before the startup deadline expires. If that never happens, DICTATR now records `startup_timeout_no_first_callback`, treats the attempt as a recoverable start failure, and hands control back to the existing reconnect path instead of hanging in a fake recording state with `captureState=never_started`.
- Removed the earlier policy that treated any Bluetooth involvement in a running route as grounds for an automatic fatal reset. While `AVAudioEngine` is still running, DICTATR now rereads the live graph formats and only reinstalls the tap if the graph is genuinely ready. The “Bluetooth route present therefore tear everything down immediately” behavior is gone. The remaining forced-reset path is the case the logs actually supported: CoreAudio has already stopped the engine, so the app must rebuild from a fresh attempt rather than mutating a dead graph.
- Replaced the retry-stage “wait until the route is stable enough, then assume success” model with advisory route-age logging only. Retries still back off, but `AppState` no longer treats elapsed time since the last route change as the primary readiness signal. The route age is now logged for forensics, while the real retry success gate is whether the restarted recorder reaches a live graph-ready state or gets its first callback. This matches the April 13 evidence where the route could look stable for seconds while the graph was still stale or the next `engine.start()` immediately triggered a fresh HFP reconfiguration.
- Added new diagnostics events specifically for the failure that had been missing from prior releases: `stale_prestart_format`, `post_start_format_ready`, `first_tap_seen`, `startup_timeout_no_first_callback`, and `retry_aborted_graph_still_stale`. Combined with the existing route-epoch, failure-incident, and heartbeat logs, the installed build should now make it obvious whether a retry died because the engine graph never became ready, because the engine was stopped by CoreAudio, or because capture actually flowed and failed later.
- Added `AVAudioRoutingArbiter` participation around recorder start and stop on supported macOS versions. This does not hardcode any headset profile, sample rate, or device name, and it does not make arbitration failure fatal. It simply gives the operating system the generic Apple-supported hint that DICTATR is about to do play-and-record audio I/O on a nearby wireless route, while logging whether the system changed the default device as a result. This preserves the repo’s “no per-headset hacks” rule while still adopting the Apple-supported route-arbitration hook that DICTATR had not been using.
- Added a parallel internal `AVCaptureSession + AVCaptureAudioDataOutput` comparison backend behind an internal backend selector. The comparison backend intentionally leaves `audioSettings` at the native-device default so the session vends the microphone’s native captured format automatically, then converts that data into the same `16000 Hz` mono float file format used by the transcription pipeline today. The shipped default remains the repaired `AVAudioEngine` path for now, but the comparison backend exists in-tree so a future switch does not start from zero again if live Bluetooth acceptance still fails.
- Added the repo’s first automated tests for recorder startup policy. The new package test target locks the readiness decision that rejects stale Bluetooth `24000/48000` graphs, accepts matching built-in `48000/48000` and Bluetooth `24000/24000` graphs, and proves that route quiet by itself is not enough to satisfy the retry success gate. These tests are intentionally pure-policy tests rather than fake `AVAudioEngine` tests, because what needed to be preserved was the decision logic, not a mocked CoreAudio implementation.
- Wrote a new canonical investigation document at [`docs/bluetooth_audio_resolution_2026-04-13.md`](/Users/hannojacobs/Documents/Code/DICTATR/docs/bluetooth_audio_resolution_2026-04-13.md) and linked both older Bluetooth postmortems to it. That document captures the exact installed-app evidence from `/Applications/DICTATR.app`, the successful AirPods sessions `235bbed3` and `f3febfa5`, the failed AirPods sessions `60f04043`, `433a2e66`, `4e7b83ba`, `bc07c691`, `0fdb2137`, `94ebc35e`, `3da20684`, and `a82f3e9e`, the corroborating failed launch `dictatr-20260413-160106-adfbc119.log`, the unified-log good/bad windows, the Apple-source conclusions, and the explicit decision rule for when to switch the production default to the capture backend instead of guessing again.
- Added the first post-ship installed-build Bluetooth success evidence to the operator record. At `2026-04-13 18:00 SAST` in [`dictatr-20260413-175125-2e38e948.log`](/Users/hannojacobs/Library/Application%20Support/DICTATR/Logs/dictatr-20260413-175125-2e38e948.log), AirPods session `c89d634e` started from a mixed route fingerprint of `inHz=24000.0|outHz=48000.0`, reached `post_start_format_ready` with a live graph of `24000/24000`, recorded with `captureState=flowing`, wrote `84630` frames, and transcribed `"Let me test. This is on my AirPods again. This is on my AirPods again."` from the installed `/Applications/DICTATR.app` build. That is the concrete evidence that the new readiness gate seems to have worked on a real AirPods start instead of looping through `captureState=never_started`.
- The correct shipped framing for `1.31` is therefore: the new recorder start/readiness model appears to have fixed the reproduced April 13 AirPods reconnect-loop case on the first installed-build retest at `18:00 SAST`, while broader Bluetooth reliability still needs more live coverage across additional devices and recovery scenarios before it should be treated as universally solved.

## 1.28
- Removed the `Hard Reset Audio` menu action and deleted the associated process-kill reset path so DICTATR no longer exposes a one-click escape hatch that terminates external Chromium/Electron audio helpers from the menu.
- Removed Developer ID as the assumed release path for DICTATR's normal ship flow. The repo now treats ad-hoc signing as the default supported distribution mode because this tool is for self-use first, and the public website/install instructions already tell users how to open an ad-hoc DMG and grant permissions locally.
- Fixed the GitHub Actions release workflow so it no longer silently falls back to `developer_id` mode on GitHub-hosted macOS runners. CI now declares `DICTATR_CODESIGN_MODE=adhoc` together with `DICTATR_SPCTL_EXPECT=rejected`, which matches the actual intended release model and stops the DMG packaging step from failing on a missing Developer ID identity.
- Fixed the installed-build dictation regression introduced by the archive-based release flow on April 13, 2026, where `/Applications/DICTATR.app` could still start, show the normal menu state, and record a WAV file full of frames, but the actual captured waveform was silent because the installed ad-hoc build had lost the microphone entitlement during packaging.
- Corrected the release signing pipeline so ad-hoc builds now embed `com.apple.security.device.audio-input=true` and no longer enable hardened runtime accidentally, while Developer ID builds still keep the hardened-runtime requirement. This directly fixes the TCC failure mode where macOS refused to prompt for microphone access because the signed app did not qualify for `kTCCServiceMicrophone`.
- Added a checked-in app entitlements file and made release verification print the embedded entitlements for both the archived bundle and the installed `/Applications/DICTATR.app`. The release scripts now fail immediately if the shipped app is missing the microphone entitlement or if ad-hoc mode accidentally carries a runtime flag again.
- Tightened release verification so the packaged build is no longer treated as correct just because `codesign --verify` passes. DICTATR now validates bundle identity, signing mode, hardened-runtime expectations, embedded microphone entitlement state, designated requirement shape, `spctl` expectations, and live launch-log evidence from the installed app.
- Fixed a verifier bug where `install-release.sh` could still reset Accessibility and fail even after the newly installed ad-hoc app had already launched with `accessibilityTrusted=yes`. Full-send verification now trusts the installed app's real launch log and only forces the Accessibility repair path when the live installed build actually reports `accessibilityTrusted=no`.
- Centralized microphone permission handling into one shared code path used by launch, onboarding, the menu-bar UI, and the recording start path. This removes the previous behavior where onboarding state could claim setup was complete even though a newly installed signed app still needed a fresh microphone grant.
- Changed the recording gate so DICTATR will not begin capture when microphone access is denied or restricted, and it now requests access up front when the state is `notDetermined`. The app reports an explicit microphone-permission problem instead of falling through into a recording session that looks successful in timing terms but produces silent audio and empty dictation.
- Added visible microphone status to the menu, alongside the existing Accessibility row, so a broken installed build is diagnosable from the app itself. The menu and onboarding now both route the user to the correct microphone permission flow or to System Settings when the app needs repair.
- Preserved raw Whisper placeholder-token logging for forensics, but changed the user-facing behavior so placeholder-only output and long silent captures are treated as microphone/capture failures instead of ordinary “No speech detected.” This keeps the diagnostic evidence while removing the misleading symptom where the dictated text simply appears to vanish.
- Verified the shipped fix on the live installed app, not just in the development build. The installed `1.28` app launched from `/Applications/DICTATR.app`, requested microphone permission successfully, transitioned to `microphoneStatus=authorized`, recorded non-zero waveform data, produced real transcription output, and pasted that transcription back into the active app.
- Verified the TCC behavior change directly in system logs. The old `tccd` failure stating that DICTATR was missing `com.apple.security.device.audio-input` is gone for the new installed build, and macOS now emits the expected microphone prompt and grant path instead of policy-denied silent capture.

## 1.30
- Expanded DICTATR’s Bluetooth/audio failure forensics from “good local breadcrumbs” into a correlated incident model that ties together `AppState`, `AudioRecorder`, and the CoreAudio route observer. The app now stamps recording attempts, recovery cycles, engine instances, failure incidents, and route epochs into the log context so a future AirPods failure can be reconstructed from one log file without manually matching timestamps across unrelated subsystems.
- Added a typed recording-failure taxonomy instead of relying on free-form user-facing strings as the only clue. Internal logs now distinguish route churn, Bluetooth HFP renegotiation, engine-start failures, converter/input-format failures, watchdog timeouts, capture stalls, duplicate failure callbacks during recovery, and retry-budget exhaustion. Support work can now key off stable reason codes instead of trying to parse prose like “Audio device changed. Reconnecting...”.
- Upgraded route diagnostics so DICTATR now records explicit route-stability timing and Bluetooth-mode inference rather than only before/after device snapshots. Route observer entries now include `routeEpoch`, `stableForMsBeforeChange`, `timeSinceAttemptStartMs`, `timeSinceEngineStartMs`, aggregate-device IDs, and a best-effort `bluetoothModeGuess` of `a2dp`, `hfp`, `mixed`, or `unknown`, plus the confidence and reasoning used for that guess. This is the evidence needed to answer whether the app retried during active Bluetooth churn instead of after the route had actually settled.
- Deepened the recorder lifecycle logs so the live app now records engine creation, input-format inspection, tap-install request/success, engine-start request/success, configuration-observer attachment/detachment, timer/watchdog teardown, force-reset start/completion, and configuration-change handling with the shared forensic context attached. When Cocoa/CoreAudio startup fails, DICTATR now logs structured `NSError` domain/code/userInfo details rather than collapsing everything into a single localized-description line.
- Added capture-cadence forensics to the real-time recording path so DICTATR can now prove whether a bad session never received tap callbacks, received callbacks but never converted data, converted data but never wrote frames, or flowed briefly and then stalled. Heartbeats and failures now carry `captureState`, `firstTapCallbackMs`, `firstNonZeroWriteMs`, callback counts, raw versus converted frame totals, dropped-buffer counts, callback-interval timing, and “time since last tap/write” fields. This is the missing evidence for distinguishing Bluetooth churn from silent capture regressions.
- Tightened recovery-policy diagnostics without changing the recovery policy itself. Retry handling now records explicit reconnect-wait start/extend/complete events, retry-budget before/after accounting, whether a failure consumed budget, whether a duplicate failure was coalesced into an existing recovery cycle, and whether the app considered the route stable before attempting the next start. The shipped behavior is still “better evidence, not a claim that Bluetooth reliability itself is fixed,” which matters because the underlying AirPods issue has not yet been proven solved.
- Added a bounded in-memory breadcrumb ring buffer plus terminal incident summaries so the installed app emits one operator-facing support bundle when a recovery cycle fails or eventually recovers. The summary includes the failure incident ID, attempt count, route-epoch span, Bluetooth-mode transitions, stable-route windows, first/final failure timing, cumulative frames written, final recommended operator action, recent route events, recent recorder lifecycle events, and retry-accounting lines. On terminal failure the app also logs the concrete support collection command `system_profiler SPAudioDataType SPBluetoothDataType`.
- Preserved the repo’s “question whether this should exist” constraint by not adding new audio-routing workarounds, reset hacks, or hidden fallback behavior. This version does not claim to repair Bluetooth microphone reliability; it deliberately ships richer observability so the next failure can be diagnosed from `/Applications/DICTATR.app` rather than from memory or guesswork. Full-send verification for `1.30` must therefore prove the installed app launches from `/Applications/DICTATR.app`, reports `version=1.30 build=1.30`, and writes the new route/attempt/incident forensic fields into `latest.log` when exercised locally.

## 1.29
- Expanded audio forensics so DICTATR now logs stable active-input and active-output identity snapshots instead of only broad route summaries. The new snapshots include device ID, UID, transport type, nominal sample rate, input/output channel counts, device-alive state, and best-effort input source metadata when CoreAudio exposes it, which makes it much easier to answer the concrete question “which microphone was DICTATR actually on?” during a failure.
- Added a dedicated CoreAudio route observer that watches default input changes, default output changes, device inventory changes, and nominal sample-rate changes on the currently active devices. When any of those events fire, DICTATR now records the previous route fingerprint, the new route fingerprint, a classified transition summary, the before/after device identities, and the full route snapshot. Device inventory dumps are still limited to inventory-change events so the log does not become unreadable during normal operation.
- Tightened microphone permission forensics so permission refreshes now record both the source of the refresh and the old/new authorization state instead of silently replacing the cached value. The permission request path now logs request start, the returned boolean grant result, and the post-prompt effective AV authorization state separately, while settings-open actions also emit an explicit diagnostic line with the current microphone status and the System Settings URL.
- Strengthened `AudioRecorder` session-start logging so each recording now captures the initial route fingerprint, active input identity, active output identity, Bluetooth-involvement flags, and the preflight engine input format together in one place. DICTATR also now records whether the route fingerprint changed between the initial preflight snapshot and the post-`engine.start()` snapshot, which is important for Bluetooth/HFP investigations where the route can mutate during startup rather than only after the session is already stable.
- Added a one-second recording heartbeat for maximum-detail forensics. During an active recording, the log now records elapsed time, engine-running state, total written frames, dropped frames, delta frames since the previous heartbeat, the current route fingerprint, the current active input/output identities, and whether Bluetooth is involved on input, output, or both. If the route fingerprint changes, the heartbeat emits the full route snapshot; if frame growth stops while recording is still nominally alive, the heartbeat marks `captureStalled=yes` so “silent but still recording” failures leave a concrete breadcrumb before the watchdog or manual stop path fires.
- Upgraded configuration-change, watchdog, stop, and force-reset diagnostics so they now carry route fingerprints and the last-known active mic context instead of only whatever snapshot happened to exist at the moment of failure. This means a Bluetooth churn incident can now be reconstructed as a sequence of route transitions and stalled/non-stalled heartbeats rather than a single isolated “config change during bluetooth route churn” message with missing before-state.
- Added a session-end forensic summary in the app state layer so DICTATR now logs route-at-start, route-at-stop, route-at-transcription-start, whether the route changed during the session, whether any zero-growth heartbeat windows occurred, and whether the captured waveform was silent despite non-zero frames. That gives one final operator-facing summary line that ties together recorder behavior and post-stop signal analysis without changing the app’s routing or recovery policy.
- Preserved the repo’s “logging only” constraint for this change. The implementation does not alter route-selection policy, Bluetooth fallback policy, retry timing, watchdog thresholds, or transcription behavior; it only adds richer observability so future weird Bluetooth, microphone, or permission bugs can be diagnosed from the installed app’s persistent `latest.log` instead of from guesswork.
- Full-send verification for this version must continue to avoid claiming that Bluetooth reliability itself is fixed. The intended shipped outcome for `1.29` is better evidence, not a product-level Bluetooth repair: the installed app should launch from `/Applications/DICTATR.app`, report `version=1.29 build=1.29`, and emit the new route-observer, permission-transition, recorder-heartbeat, and session-summary diagnostics when the audio path is exercised locally.
